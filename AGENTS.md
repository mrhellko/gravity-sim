# Инструкции для Codex

## Язык

- Всегда отвечать пользователю на русском языке, если пользователь явно не попросил другой язык.

## Цель проекта

Проект `gravity-sim` - браузерный 3D-симулятор гравитации, доступный по адресу `https://g.order.mrhellko.ru`.

Основной домен `order.mrhellko.ru` уже используется другим проектом на этом VPS. Нельзя менять его поведение, удалять его конфиги, контейнеры, сертификаты, переменные окружения или порты без явного запроса пользователя.

## Архитектурные правила

- Серверная часть должна быть минимальной: раздача статических файлов и, при необходимости, reverse proxy.
- Все тяжелые вычисления выполняются на стороне пользователя в браузере. При наличии WebGPU использовать GPU пользователя для расчетов, при отсутствии WebGPU использовать CPU/Web Worker fallback.
- Для 3D использовать WebGPU-first подход с WebGL/CPU fallback. Базовый выбор для MVP: Three.js, с выделенным backend-слоем для физики.
- Физика MVP: ньютоновская гравитация без ОТО.
- Расчеты не должны зависеть от мощности VPS.
- Если в проекте появляется серверный runtime или контейнер, он должен иметь жесткие лимиты CPU/RAM/PID и не должен выполнять физические расчеты.
- При добавлении деплоя изолировать сервис для `g.order.mrhellko.ru` от существующего проекта `order.mrhellko.ru`.

## Ограничения VPS и деплоя

- Не занимать порты, которые уже используются существующими проектами.
- Не перезаписывать глобальные конфиги Caddy/Nginx/Traefik/Docker Compose без предварительного чтения текущей конфигурации.
- Для нового домена использовать отдельный site block / route / compose service.
- Перед изменениями инфраструктуры обязательно проверить текущие контейнеры, слушающие порты и конфиги прокси.
- Предпочитать статический build, который можно отдавать через Caddy/Nginx без Node.js runtime.
- Для Docker Compose задавать resource limits (`cpus`, `mem_limit`, `pids_limit`, restart policy) для любых сервисов gravity-sim.
- Не добавлять API-эндпоинты, которые принимают тяжелые расчетные задачи от клиента.

## Качество реализации

- UI должен быть рабочим приложением, а не лендингом.
- Камера управляется мышью: вращение, приближение, панорамирование.
- В интерфейсе должны быть базовые элементы управления симуляцией: старт, пауза, продолжить, сброс.
- Показывать FPS и ключевые параметры симуляции.
- Состояние начальной сцены должно быть воспроизводимым: сброс возвращает объекты к исходным позициям/скоростям/массам.
- Для численной физики использовать стабильный интегратор для MVP, минимум velocity Verlet или leapfrog. Архитектура должна позволять выполнять одинаковую модель через CPU backend и WebGPU compute backend.
- Не добавлять сложную абстракцию раньше необходимости.

## Субагенты

- Подключать субагентов только для узких параллельных задач, когда есть готовая структура проекта или отдельная исследовательская задача.
- Перед подключением субагента сверяться с `.agents/subagents.md`.
- Основной агент отвечает за финальную интеграцию и не должен принимать несовместимые изменения от разных субагентов без проверки.
