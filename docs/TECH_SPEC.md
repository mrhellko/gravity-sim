# Техническое задание: Gravity Sim

## 1. Назначение

`Gravity Sim` - браузерное 3D-приложение для интерактивной симуляции гравитационного взаимодействия тел по закону Ньютона.

Приложение должно быть доступно по адресу:

```text
https://g.order.mrhellko.ru
```

Существующий проект на `https://order.mrhellko.ru` не должен быть затронут.

## 2. Ключевые ограничения

- VPS слабый и не должен выполнять расчеты физики.
- Рендер и расчеты выполняются в браузере пользователя.
- Сервер используется только для раздачи статического фронтенда и, при необходимости, проксирования.
- Любой серверный runtime, если он появится, должен иметь жесткие ограничения CPU/RAM/PID и не должен выполнять симуляцию.
- Деплой должен быть изолирован от существующих проектов на VPS.
- Физическая модель MVP: классическая ньютоновская гравитация, без общей теории относительности.

## 3. Целевая платформа

- Современные desktop-браузеры: Chrome, Edge, Firefox, Safari.
- Основной сценарий управления: мышь и клавиатура.
- WebGPU является целевым ускоренным backend для расчетов и, по возможности, рендера.
- WebGL/CPU fallback обязателен, потому что поддержка WebGPU зависит от браузера, ОС, драйверов и настроек пользователя.
- Приложение должно определять доступность WebGPU через `navigator.gpu` и показывать текущий backend в UI.

## 4. Физическая модель

Для каждой пары тел `i` и `j` рассчитывается сила:

```text
F = G * m_i * m_j / r^2
```

Где:

- `G` - настраиваемая гравитационная постоянная симуляции.
- `m_i`, `m_j` - массы тел.
- `r` - расстояние между центрами масс.

Для устойчивости расчета используется softening factor:

```text
F = G * m_i * m_j / (r^2 + e^2)
```

Где `e` - малое значение сглаживания, предотвращающее бесконечные ускорения при очень малых расстояниях.

MVP-интегратор: velocity Verlet или leapfrog. Простой Euler использовать только для временного прототипа и заменить до первого рабочего релиза.

Расчетная архитектура:

- `PhysicsBackend` как общий интерфейс;
- `CpuPhysicsBackend` для базовой версии и fallback;
- `WorkerPhysicsBackend` для CPU-расчетов без блокировки UI;
- `WebGpuPhysicsBackend` для расчетов на GPU пользователя;
- одинаковые входные данные и одинаковая семантика результата для всех backend.

WebGPU backend должен использовать WGSL compute shader и GPU buffers для позиций, скоростей, масс и ускорений.

## 5. Объекты симуляции

Пользователь должен уметь:

- добавлять тела из набора примитивов;
- выбирать тело кликом левой кнопки мыши в 3D-сцене;
- видеть визуальное выделение выбранного тела;
- редактировать название тела;
- получать автоматически сгенерированное название при создании тела;
- выбирать цвет тела через RGB-поля и палитру;
- задавать массу тела;
- задавать массу в обычной и экспоненциальной записи, например `5.972e24`;
- задавать начальное положение;
- задавать начальную скорость;
- фиксировать тело на месте через `pinned/static` флаг;
- выбирать/перемещать тело в 3D-сцене;
- удалять тело;
- сбрасывать симуляцию к начальному состоянию.

Инспектор выбранного тела должен поддерживать:

- `name` - текстовое название, автогенерация по словарю с возможностью ручного изменения;
- `color` - RGB/hex значение и выбор из палитры;
- `mass` - масса в экспоненциальной записи;
- `x`, `y`, `z` - координаты;
- `vx`, `vy`, `vz` - скорость;
- `pinned` - флаг фиксации тела;
- `radius` или визуальный размер;
- readonly-поля для текущего ускорения `ax`, `ay`, `az`, если ускорение уже рассчитано;
- кнопку фокуса камеры на выбранном теле;
- кнопку удаления тела.

Поведение `pinned/static` тела:

- тело остается в заданных координатах;
- скорость и ускорение тела принудительно равны `0`;
- другие тела продолжают испытывать его гравитационное влияние;
- само pinned-тело не изменяет позицию под действием других тел;
- в инспекторе координаты pinned-тела можно менять вручную только когда симуляция остановлена или на паузе;
- при включении `pinned` текущая скорость тела сбрасывается в `0`;
- reset восстанавливает исходное pinned-состояние.

Минимальный набор объектов для MVP:

- сфера;
- тяжелое центральное тело;
- легкое орбитальное тело;
- готовый пресет системы из нескольких тел.

Расширения после MVP:

- кубы/астероиды как визуальные формы;
- столкновения;
- слияние тел;
- трассы орбит;
- импорт/экспорт сцены.

## 6. Управление симуляцией

Обязательные команды:

- старт;
- пауза;
- продолжить;
- сброс к начальному состоянию;
- очистка сцены;
- изменение скорости времени;
- изменение `G`;
- изменение шага расчета или качества симуляции.
- ускорение и замедление симуляции через time scale;
- изменение fixed timestep в UI с безопасными пределами;
- отдельное отображение render FPS и simulation step.

Сброс должен восстанавливать исходные:

- позиции;
- скорости;
- массы;
- параметры сцены;
- состояние таймера симуляции.
- выбранный timestep и time scale.

## 7. Управление камерой

Пользователь должен управлять камерой мышью:

- вращение вокруг сцены;
- приближение/отдаление колесом;
- панорамирование;
- фокус на выбранном объекте.

Для MVP допустимо использовать `OrbitControls` из Three.js.

Система координат:

- основная рабочая плоскость симуляции - `XOY`;
- ось `Z` направлена вверх;
- стартовые пресеты и новые тела по умолчанию создаются в плоскости `XOY` с `Z = 0`;
- сетка сцены отображает плоскость `XOY`.

Фокус камеры:

- команда `Focus` работает как переключатель follow-режима;
- когда follow включен, камера продолжает следить за выбранным телом;
- при отсутствии выбранного тела follow-режим выключается.

## 8. Интерфейс

На экране должны быть:

- 3D-сцена как основной рабочий холст;
- панель управления симуляцией;
- панель добавления объектов;
- инспектор выбранного тела;
- FPS;
- счетчик объектов;
- текущее значение `G`;
- текущая скорость времени;
- текущий fixed timestep;
- текущий масштаб сцены;
- статус симуляции: stopped/running/paused.
- инспектор выбранного тела с редактированием имени, цвета, массы, координат и скоростей;
- палитра быстрых цветов;
- отображение шлейфов/траекторий тел цветом соответствующего тела.
- список существующих тел;
- выбор тела кликом по строке списка;

Интерфейс должен быть утилитарным, без лендинга и маркетингового первого экрана.

Редактирование параметров тела:

- инспектор скрыт, если тело не выбрано;
- параметры тела доступны для редактирования только когда симуляция остановлена или на паузе;
- изменение параметров тела не должно сбрасывать elapsed time симуляции;
- во время выполнения симуляции поля `x`, `y`, `z`, `vx`, `vy`, `vz`, `ax`, `ay`, `az` должны отображать live runtime-значения выбранного тела;
- выбранное тело подсвечивается габаритным контейнером с короткими ребрами у углов, а не кольцом.

Дополнительные требования к визуализации:

- показывать масштаб сцены, например длину grid cell или scale bar;
- поддерживать включение/выключение шлейфов;
- ограничивать длину шлейфов, чтобы не расходовать память без контроля;
- шлейф тела отображать полосой или линией цвета тела;
- подсвечивать выбранное тело обводкой, halo или marker-элементом;
- показывать предупреждение, если выбранный timestep может ухудшить стабильность симуляции.

## 9. Производительность

MVP должен комфортно работать с десятками тел на обычном клиентском устройстве.

Целевые ориентиры:

- 60 FPS для малых сцен;
- стабильная работа при 100-300 телах, если устройство пользователя позволяет;
- расчет физики не должен блокировать UI сильнее необходимого.

План оптимизаций:

- сначала корректная CPU-версия как эталон;
- затем WebGPU compute backend;
- затем Web Worker fallback для браузеров без WebGPU;
- затем spatial partitioning / Barnes-Hut для большого числа тел;
- затем оптимизация передачи данных между CPU и GPU.

WebGPU не отменяет CPU-эталон: он нужен для тестов, отладки физики и fallback.

## 10. Деплой

Цель:

```text
g.order.mrhellko.ru -> gravity-sim frontend
order.mrhellko.ru -> существующий проект без изменений
```

Перед деплоем обязательно проверить:

- текущий reverse proxy;
- активные контейнеры;
- слушающие порты;
- текущие DNS/сертификаты;
- конфигурацию существующего `order.mrhellko.ru`.

Предпочтительный формат деплоя:

- статический frontend build;
- отдельный Docker Compose service или отдельная директория статических файлов;
- отдельный vhost/site block для `g.order.mrhellko.ru`;
- TLS через текущий reverse proxy, если он уже используется.

Защита VPS:

- не запускать серверный расчет физики;
- не добавлять backend API для симуляции;
- отдавать frontend как статику, если это возможно;
- для контейнера раздачи статики задавать лимиты ресурсов;
- пример целевых лимитов для легкого static service: `cpus: 0.25`, `mem_limit: 128m`, `pids_limit: 64`;
- для reverse proxy не менять лимиты существующих проектов без отдельного запроса;
- добавить healthcheck только легкого типа, без расчетной нагрузки;
- перед публикацией проверить потребление ресурсов командой `docker stats` или аналогом.

## 11. Критерии готовности MVP

MVP считается готовым, когда:

- приложение открывается в браузере;
- сцена рендерится в 3D;
- камера вращается мышью;
- можно добавить минимум два тела с массой;
- тела притягиваются по закону Ньютона;
- можно менять `G`;
- можно запустить, поставить на паузу, продолжить и сбросить симуляцию;
- отображается FPS;
- отображается активный backend физики: CPU, Worker или WebGPU;
- build можно раздать как статику;
- деплой не ломает `order.mrhellko.ru`.
