# Конфигурация субагентов

Этот файл описывает роли субагентов для проекта `gravity-sim`. Субагентов подключать только когда появляется реальная параллельная работа; финальные архитектурные решения и интеграцию держит основной агент.

## Общие правила

- Все ответы и промежуточные отчеты должны быть на русском языке.
- Не менять существующий проект `order.mrhellko.ru`.
- Не выполнять деплой и инфраструктурные изменения без отдельной команды пользователя.
- Если появляется сервис на VPS, требовать лимиты CPU/RAM/PID и запрет серверных расчетов физики.
- Не принимать архитектурные решения, конфликтующие с `AGENTS.md`, `docs/TECH_SPEC.md` и `docs/IMPLEMENTATION_PLAN.md`.
- Каждый субагент должен работать в своей зоне ответственности и возвращать короткий отчет: что сделано, какие файлы изменены, какие риски остались, как проверено.
- Если задача требует изменения общих контрактов, сначала вернуть предложение основному агенту, а не менять контракт молча.

## physics-webgpu

Назначение: физическая модель, численная интеграция, WebGPU compute.

Подключать начиная с этапа 2 плана реализации.

Зона ответственности:

- интерфейс `PhysicsBackend`;
- `CpuPhysicsBackend` как эталон корректности;
- `WebGpuPhysicsBackend`;
- WGSL compute shader;
- сравнение CPU/WebGPU результатов;
- тесты физики;
- численная устойчивость.

Не отвечает за:

- визуальный дизайн UI;
- деплой;
- настройку reverse proxy.

Ожидаемые артефакты:

- код backend физики;
- минимальные тесты на ускорение, сохранение состояния и reset;
- отчет о допустимой численной погрешности CPU vs WebGPU;
- список ограничений WebGPU backend.

Критерии приемки:

- малые сцены дают близкий результат на CPU и WebGPU;
- при отсутствии WebGPU есть корректный fallback;
- расчет не блокирует UI сильнее, чем CPU baseline.

## scene-ui

Назначение: 3D-сцена, управление камерой, интерфейс пользователя.

Подключать начиная с этапа 1 плана реализации.

Зона ответственности:

- Three.js renderer, scene, camera, lights;
- OrbitControls;
- full-screen canvas;
- FPS overlay;
- панели управления симуляцией;
- список тел и инспектор выбранного тела;
- raycasting и выбор объектов;
- базовое 3D-перемещение объектов.

Не отвечает за:

- физическую корректность интегратора;
- WGSL compute;
- production-деплой.

Ожидаемые артефакты:

- рабочая сцена;
- UI для start/pause/resume/reset;
- UI для `G`, скорости времени и активного backend;
- проверка, что текст и элементы управления не перекрываются.

Критерии приемки:

- сцена открывается как приложение, не как лендинг;
- камера управляется мышью;
- объект можно выбрать;
- FPS и backend видны пользователю.

## deploy-infra

Назначение: безопасная публикация на `g.order.mrhellko.ru`.

Подключать только на этапе 8 плана реализации или по отдельному запросу пользователя.

Зона ответственности:

- инвентаризация текущего VPS;
- проверка активных контейнеров;
- проверка слушающих портов;
- изучение текущего reverse proxy;
- отдельный deployment target для `g.order.mrhellko.ru`;
- resource limits для нового сервиса;
- проверка отсутствия серверных расчетных endpoints;
- проверка, что `order.mrhellko.ru` не изменился.

Не отвечает за:

- WebGPU физику;
- UI;
- изменение существующего проекта `order.mrhellko.ru`.

Ожидаемые артефакты:

- план деплоя перед изменениями;
- список затрагиваемых файлов/сервисов;
- команды проверки до и после;
- rollback plan.
- лимиты ресурсов и команды проверки потребления.

Критерии приемки:

- `order.mrhellko.ru` продолжает отвечать как раньше;
- `g.order.mrhellko.ru` обслуживает gravity-sim;
- изменения изолированы.
- новый сервис ограничен по CPU/RAM/PID.

## qa-verification

Назначение: проверки, тесты, сборка, браузерная валидация.

Подключать после появления первого рабочего каркаса и перед каждым крупным этапом завершения.

Зона ответственности:

- unit-тесты физики;
- build checks;
- браузерные проверки;
- проверка fallback без WebGPU;
- проверка basic FPS/backend UI;
- smoke-тесты деплоя.

Не отвечает за:

- самостоятельный выбор архитектуры;
- инфраструктурные изменения без задачи deploy-infra.

Ожидаемые артефакты:

- список выполненных проверок;
- найденные баги с воспроизведением;
- тестовые сценарии;
- остаточные риски.

Критерии приемки:

- production build проходит;
- базовая симуляция воспроизводима;
- fallback path проверен;
- известные ограничения описаны.

## Когда подключать

- Этап 1: `scene-ui`.
- Этап 2: `physics-webgpu` для CPU backend.
- Этап 3: `physics-webgpu` для WebGPU backend, `qa-verification` для сравнения результатов.
- Этапы 4-5: `scene-ui`, при необходимости `physics-webgpu`.
- Этап 6: `physics-webgpu` и `qa-verification`.
- Этап 8: `deploy-infra`, затем `qa-verification`.
